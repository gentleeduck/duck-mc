# duck-benchmarks

Five recorded bench runs from the dmc optimisation work, in chronological
order. Each phase folder holds the raw `bench.json` plus the three plot
SVGs (`scale.svg`, `table.svg`, `throughput.svg`) generated by
`dmc-core/examples/bench.rs` against the same fixture set on the same
host (32 CPU, x86_64, linux).

## Headline (1000 files, median ms, lower is better)

| phase | native | sidecar+gfm | sidecar+pretty-code | sidecar+kitchen-sink | velite+gfm | velite+kitchen-sink |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| [1 baseline](phase-1-baseline)              | 10.65 | 404.86 | 806.87 | 2652.43 | 5984.50 | 1375.06 |
| [2 single-walk](phase-2-single-walk)        | 11.92 | 446.19 | 854.87 | 2666.42 | 6135.08 | 1398.11 |
| [3 native-syntect](phase-3-native-syntect)  | 47.42 | 507.41 | 492.93 |  885.74 | 6785.13 | 1446.52 |
| [4 confirm-run](phase-4-confirm-run)        | 44.69 | 462.33 | 466.37 | 1187.90 | 5910.21 | 1368.20 |
| [5 plugin-gate](phase-5-plugin-gate)        | 44.73 |  46.01 |  44.94 |  144.77 | 5934.00 | 1381.46 |
| [6 correctness-cache](phase-6-correctness-cache) | 55.19 |  51.97 |  50.65 |  168.93 | 6110.71 | 1427.16 |

`native` jumps phase 2 -> 3 because the syntect highlighter moved
in-process (more native work, less sidecar work). `sidecar+*` collapse
phase 4 -> 5 because the plugin gate stops shipping native-owned plugins
to the sidecar. Phase 6 is a deliberate ~10-19% trade: correctness
work (CommonMark §4.5/§6.1 fence + inline rules, lowercase-JSX depth,
real span file paths, structured `BuildReport.diagnostics`) plus a
two-layer cache (preMdx mirror manifest + native compile cache that
now survives `clean: true`). The cold bench above doesn't see the
caches; the warm-build delta is **34 s → 3.3 s** end-to-end on a 370-
mdx consumer corpus (see phase-6 README).

## Speedups vs velite at 1000 files (phase 5)

| compared | velite ms | dmc ms | speedup |
| --- | ---: | ---: | ---: |
| native vs velite+gfm                | 5934.00 | 44.73   | 132x |
| native vs velite+kitchen-sink       | 1381.46 | 44.73   |  30.9x |
| sidecar+kitchen-sink vs velite+ks   | 1381.46 | 144.77  |   9.5x |
| sidecar+gfm vs velite+gfm           | 5934.00 | 46.01   | 129x |

Velite numbers stay near constant across phases (sanity check that the
host and fixtures did not drift).

## Optimisation track (what changed phase to phase)

| phase | change | source commit |
| --- | --- | --- |
| 1 | initial native pipeline; sidecar runs every JS plugin verbatim | pre `78a9400` |
| 2 | single-walk pipeline (NodeSink + Walker + Accumulator) merges lex + parse + transform + emit into one pass | `78a9400` |
| 3 | syntect bundled natively; pretty-code highlighting runs in Rust, sidecar no longer needs `rehype-pretty-code` work | `b2f7a03`, `1aa91eb` |
| 4 | confirm run on the same configuration; numbers within noise of phase 3 | between `1aa91eb` and `8c8965f` |
| 5 | plugin gate widening: sidecar payload strips every native-owned remark / rehype plugin (gfm, pretty-code, katex, slug, autolink-headings); sidecar becomes a no-op forwarder when the listed plugins all match natives | `8c8965f` |

## How to reproduce

```sh
cargo run --release -p dmc-core --features pretty-code --example bench
```

Outputs land in `dmc-core/tmp/` (gitignored). To compare against this
folder, copy the fresh `bench.json` + `*.svg` into a new
`duck-benchmarks/phase-N-<label>/` and add a README to match.

## What is in each folder

| file | content |
| --- | --- |
| `bench.json` | per-fixture sample arrays plus 10 / 100 / 1000-file size sweeps for every variant (`native`, three `sidecar+*` variants, two `velite+*` variants) |
| `scale.svg` | log-log scaling chart over file count |
| `table.svg` | tabular rendering of the median numbers |
| `throughput.svg` | files/sec rendering for the size sweep |
| `README.md` | what changed in this phase + phase-local table |

## Honest caveats

- Each median is over 200 samples (per-file variants) or 10 samples (size
  sweep). Small movements between phases 3 and 4 sit inside the
  stddev band; treat them as noise.
- Velite invocations shell out to the velite CLI; the per-call cost
  includes Node startup, so the baseline favours dmc on small fixture
  counts. The 1000-file column is the most honest comparison.
- The fixture set lives in `examples/nextjs/content/` and changes over
  time. Re-running phase 1 today on current fixtures will not reproduce
  the phase 1 numbers; the phase numbers are a *historical* track of
  what the host saw at that commit, not a re-runnable benchmark.
